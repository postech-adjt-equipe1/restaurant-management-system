Visão Geral do Projeto

O projeto é um backend REST em Spring Boot para gestão de usuários de um sistema de restaurantes, desenvolvido como entregável do Tech Challenge Fase 1 da POSTECH FIAP. O sistema suporta dois tipos de usuário — Dono de Restaurante (OWNER) e Cliente (CUSTOMER) — e expõe operações de CRUD, troca de senha e validação de login.

Objetivo
  • Cadastrar e gerenciar usuários (CRUD completo)
  • Troca de senha com validação BCrypt
  • Validação de login (autenticação simples)
  • API documentada via Swagger/OpenAPI 3
  • Ambiente 100% containerizado com Docker
Entregáveis
  • Código-fonte no GitHub
  • Docker Compose funcional
  • Collection Postman (pendente)
  • Documentação Swagger em /swagger-ui.html
  • README detalhado

Arquitetura em Camadas

A aplicação segue o padrão arquitetura em camadas (Layered Architecture), onde cada camada tem uma responsabilidade única e bem definida. A comunicação entre camadas é sempre de cima para baixo — a camada superior depende da inferior, nunca o contrário.

HTTP Client
Postman / Browser / Frontend — faz requisições HTTP (GET, POST, PUT, DELETE, PATCH)
↓ JSON sobre HTTP ↓
Controller
Recebe requisições, valida @Valid nos DTOs, delega para o Service, retorna ResponseEntity
↓ chama ↓
Service
Contém toda a lógica de negócio: unicidade, BCrypt, regras de atualização, exceções customizadas
↓ chama ↓
Repository
Interface Spring Data JPA — abstrai acesso ao banco. Queries derivadas por nome de método
↓ SQL gerado pelo Hibernate ↓
Model (Entity)
Entidades JPA mapeadas para tabelas SQL. User @Entity + Address @Embeddable
↓ JDBC ↓
PostgreSQL
Banco de dados relacional — schema criado e versionado pelo Flyway

Camada de Domínio — Model

3.1 Entidade User

A entidade User é o coração do domínio. Ela representa um usuário do sistema e é mapeada diretamente para a tabela users no banco de dados.

src/main/java/br/com/fiap/restaurante/model/User.java
@Entity @Table(name = "users") @Getter @Setter @NoArgsConstructor @AllArgsConstructor @Builder public class User { @Id @GeneratedValue(strategy = GenerationType.IDENTITY) private Long id; @NotBlank @Size(max = 100) private String nome; @NotBlank @Email @Size(max = 150) @Column(unique = true, nullable = false) private String email; @NotBlank @Size(min = 3, max = 50) @Column(unique = true, nullable = false) private String login; @NotBlank private String senha; // sempre armazenada como hash BCrypt @Column(nullable = false) private LocalDateTime dataUltimaAlteracao; @NotNull @Enumerated(EnumType.STRING) @Column(nullable = false) private UserType tipo; @NotNull @Valid @Embedded private Address endereco; @PrePersist public void prePersist() { dataUltimaAlteracao = LocalDateTime.now(); } @PreUpdate public void preUpdate() { dataUltimaAlteracao = LocalDateTime.now(); } }

Explicação das anotações usadas em User

AnotaçãoCamadaO que faz
@EntityJPAMarca a classe como entidade gerenciada pelo Hibernate
@Table(name="users")JPADefine o nome exato da tabela no banco (sem essa anotação usaria "user", palavra reservada em SQL)
@IdJPADefine o campo como chave primária
@GeneratedValue(IDENTITY)JPADelega a geração do ID para o banco (BIGSERIAL/auto-increment)
@Column(unique, nullable)JPARefina o mapeamento da coluna: unicidade e NOT NULL no DDL
@Enumerated(STRING)JPAPersiste o enum como texto ("OWNER"/"CUSTOMER") em vez de número ordinal
@EmbeddedJPAInstrui o Hibernate a incorporar os campos de Address na mesma tabela
@ValidBean ValidationCascateia a validação para dentro do objeto Address embutido
@PrePersist / @PreUpdateJPA LifecycleCallbacks executados antes de INSERT e UPDATE — atualiza dataUltimaAlteracao automaticamente
@NotBlankBean ValidationCampo não pode ser null, vazio ou só espaços
@EmailBean ValidationValida formato de e-mail
@Size(min, max)Bean ValidationControla tamanho mínimo e máximo da String

Camada de Domínio — Model (cont.)

3.2 Value Object Address (@Embeddable)

Address não é uma entidade independente — é um value object embutido dentro de User. Isso significa que não possui tabela própria: seus campos são colunas da tabela users.

💡
@Embeddable vs @Entity: Uma entidade tem identidade própria (id) e tabela própria. Um embeddable não tem id — ele vive dentro de outra entidade. É ideal para dados que não fazem sentido isolados, como endereços.
src/main/java/br/com/fiap/restaurante/model/Address.java
@Embeddable @Builder @NoArgsConstructor @AllArgsConstructor public class Address { @NotBlank private String logradouro; @NotBlank private String numero; private String complemento; // nullable — campo opcional @NotBlank private String bairro; @NotBlank private String cidade; @NotBlank private String estado; @NotBlank @Size(max = 9) @Pattern(regexp = "\\d{5}-?\\d{3}") // aceita "12345-678" ou "12345678" private String cep; }

Como os campos de Address ficam no banco

✅
Como Address é @Embeddable, o Hibernate mapeia cada campo diretamente como coluna de users. Não há JOIN — todos os dados de um usuário ficam em uma única linha da tabela.

3.3 Enum UserType

O enum diferencia os dois perfis de usuário do sistema. É persistido como texto graças ao @Enumerated(EnumType.STRING) na entidade User.

public enum UserType { OWNER, // Dono de Restaurante CUSTOMER // Cliente }
⚠️
Por que STRING e não ORDINAL? Com EnumType.ORDINAL o valor 0 ou 1 é salvo. Se um novo valor for inserido no meio do enum no futuro, todos os registros existentes ficam com valores errados. Com EnumType.STRING, o texto "OWNER"/"CUSTOMER" é salvo — resistente a mudanças de ordem.

Repositório — UserRepository

O repositório é a camada responsável pelo acesso ao banco de dados. Com Spring Data JPA, não é necessário escrever SQL manualmente para operações simples — basta declarar uma interface que estende JpaRepository e o Spring gera a implementação em tempo de execução.

src/main/java/br/com/fiap/restaurante/repository/UserRepository.java
public interface UserRepository extends JpaRepository<User, Long> { Optional<User> findByEmail(String email); Optional<User> findByLogin(String login); boolean existsByEmail(String email); boolean existsByLogin(String login); }

O que JpaRepository já fornece gratuitamente

Método herdadoSQL geradoUso previsto
save(user)INSERT ou UPDATECadastro e atualização de usuário
findById(id)SELECT ... WHERE id = ?Buscar usuário por ID
deleteById(id)DELETE ... WHERE id = ?Excluir usuário
existsById(id)SELECT count(*) WHERE id = ?Verificar existência antes de operar
findAll()SELECT * FROM usersListar todos (uso interno/admin)

Queries derivadas por nome de método

O Spring Data JPA interpreta o nome do método e gera o SQL automaticamente. Não há implementação manual.

Método declaradoSQL gerado pelo SpringRetorno
findByEmail(email) SELECT * FROM users WHERE email = ? Optional<User> — vazio se não encontrado
findByLogin(login) SELECT * FROM users WHERE login = ? Optional<User> — usado na validação de login
existsByEmail(email) SELECT count(*) > 0 WHERE email = ? boolean — mais eficiente que findBy para checar unicidade
existsByLogin(login) SELECT count(*) > 0 WHERE login = ? boolean — verificação de duplicidade no cadastro
💡
Por que Optional e não User diretamente? Retornar Optional<User> força o código chamador a tratar explicitamente o caso de "não encontrado", eliminando NullPointerException e tornando o código mais seguro e legível. O Service deve chamar .orElseThrow(() -> new UserNotFoundException(...)).

Migrations com Flyway

O Flyway é responsável por criar e evoluir o schema do banco de dados de forma controlada e versionada. Em vez de deixar o Hibernate criar as tabelas automaticamente (ddl-auto: create), usamos scripts SQL explícitos — o que garante controle total sobre o schema e evita surpresas em produção.

Como funciona o Flyway

1. Aplicação sobe → Flyway escaneia classpath:db/migration/
2. Lê todos os arquivos no padrão V{versão}__{descrição}.sql
3. Consulta a tabela interna flyway_schema_history no banco
4. Executa apenas os scripts ainda não aplicados (novos)
5. Registra cada execução com checksum na tabela de histórico
6. Aplicação continua a inicialização normalmente

Convenção de nomenclatura dos arquivos

V1__create_users.sql ← V + versão + __ (dois underscores) + descrição V2__add_index_email.sql V3__add_column_phone.sql
⚠️
Scripts Flyway jamais devem ser alterados após serem executados. O Flyway grava o checksum de cada arquivo. Se o conteúdo mudar, ele lança erro na próxima subida. Para corrigir algo, sempre crie um novo arquivo V2__, V3__...

Script V1 — Criação da tabela users

src/main/resources/db/migration/V1__create_users.sql
-- V1: Criação da tabela de usuários CREATE TABLE users ( id BIGSERIAL PRIMARY KEY, nome VARCHAR(100) NOT NULL, email VARCHAR(150) NOT NULL UNIQUE, login VARCHAR(50) NOT NULL UNIQUE, senha VARCHAR(255) NOT NULL, -- hash BCrypt (60 chars) data_ultima_alteracao TIMESTAMP NOT NULL, tipo VARCHAR(20) NOT NULL, -- OWNER ou CUSTOMER -- Campos do Address (@Embedded) logradouro VARCHAR(200) NOT NULL, numero VARCHAR(20) NOT NULL, complemento VARCHAR(100), -- nullable bairro VARCHAR(100) NOT NULL, cidade VARCHAR(100) NOT NULL, estado VARCHAR(2) NOT NULL, -- sigla UF (SP, RJ...) cep VARCHAR(9) NOT NULL -- formato: XXXXX-XXX );

Pontos importantes do script

  • BIGSERIAL é o tipo PostgreSQL para Long auto-increment (equivale a BIGINT + sequência automática)
  • senha VARCHAR(255) — o hash BCrypt tem sempre 60 caracteres, mas 255 dá margem para outros algoritmos futuros
  • tipo VARCHAR(20) — armazena o texto do enum ("OWNER" ou "CUSTOMER")
  • estado VARCHAR(2) — apenas a sigla da UF (SP, RJ, MG...)
  • Os campos de endereço fazem parte da mesma tabela (sem JOIN) por causa do @Embedded

Segurança — SecurityConfig

A configuração de segurança tem dois objetivos nesta fase do projeto: (1) disponibilizar o BCryptPasswordEncoder como bean para criptografar senhas, e (2) desabilitar a autenticação HTTP padrão do Spring Security para que os endpoints fiquem acessíveis.

src/main/java/br/com/fiap/restaurante/config/SecurityConfig.java
@Configuration @EnableWebSecurity public class SecurityConfig { @Bean public PasswordEncoder passwordEncoder() { return new BCryptPasswordEncoder(); } @Bean public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception { http .csrf(AbstractHttpConfigurer::disable) .authorizeHttpRequests(auth -> auth .anyRequest().permitAll() ); return http.build(); } }

Por que isso é necessário?

Ao incluir spring-boot-starter-security no projeto, o Spring Security automaticamente protege todos os endpoints com autenticação HTTP Basic. Como o projeto implementa autenticação própria via POST /usuarios/login, a proteção padrão é desativada intencionalmente.

ConfiguraçãoEfeitoMotivação
BCryptPasswordEncoder @Bean Disponibiliza o encoder no container Spring O Service pode injetar e usar para hash de senhas
.csrf(disable) Desabilita proteção CSRF APIs REST stateless não usam cookies de sessão, tornando CSRF irrelevante
.anyRequest().permitAll() Libera todos os endpoints sem autenticação HTTP Autenticação é tratada na camada de negócio via POST /login

Como o BCrypt será usado pelo Service (a implementar)

@Service public class UserService { private final PasswordEncoder passwordEncoder; // injetado do SecurityConfig // Cadastro — criptografar antes de salvar user.setSenha(passwordEncoder.encode(request.getSenha())); // Login — comparar hash armazenado com senha informada passwordEncoder.matches(senhaInformada, user.getSenha()); }
⚠️
BCrypt é unidirecional. Não é possível "descriptografar" uma senha BCrypt. Para validar o login, usa-se passwordEncoder.matches(raw, hash) — que recalcula e compara o hash, sem nunca expor a senha original.

Infraestrutura Docker

9.1 Dockerfile — Build multi-stage

O Dockerfile usa multi-stage build: o primeiro estágio compila o código com Maven e JDK, o segundo estágio copia apenas o JAR gerado para uma imagem mínima com JRE. Isso resulta em uma imagem final significativamente menor.

Stage 1 — Build
  • Imagem base: maven:3.9.6-eclipse-temurin-17
  • Copia o pom.xml e baixa dependências (cache de camada Docker)
  • Copia o src/ e executa mvn package -DskipTests
  • Gera o JAR em /app/target/*.jar
Stage 2 — Runtime
  • Imagem base: eclipse-temurin:17-jre-alpine (mínima)
  • Cria usuário não-root appuser por segurança
  • Copia o JAR do Stage 1
  • Expõe porta 8080
  • Executa com java -jar app.jar

9.2 docker-compose.yml

Define dois serviços: o banco de dados PostgreSQL e a aplicação Spring Boot.

ServiçoImagemPortaDetalhe importante
postgres postgres:16-alpine 5432:5432 Healthcheck com pg_isready — a app só sobe após o banco estar pronto
app Build local (Dockerfile) 8080:8080 depends_on: postgres: condition: service_healthy — evita falhas de conexão na inicialização

Fluxo de inicialização

docker compose up --build
→ Docker constrói a imagem da app (Stage 1: Maven build → Stage 2: JRE)
→ Sobe o container postgres e aguarda healthcheck (pg_isready)
→ Após postgres healthy, sobe o container app
→ Spring Boot inicia → Flyway executa V1__create_users.sql
→ Hibernate valida schema com ddl-auto: validate
→ API disponível em http://localhost:8080

Comandos úteis

# Subir tudo (build + banco + app) docker compose up --build # Subir em background docker compose up --build -d # Parar e remover containers docker compose down # Parar e remover containers + volume do banco (reset total) docker compose down -v # Ver logs da aplicação docker compose logs -f app
